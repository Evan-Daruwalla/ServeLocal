## 2026-07-02 — Project inception  [reconstructed]

**WHAT:** Initial ServeLocal volunteer-platform codebase committed (`Initial commit: ServeLocal
volunteer platform`), followed by `Add README`. A zero-dependency Node.js web app: students find and
track community-service opportunities; orgs post them and verify hours.

**WHY:** Personal/portfolio full-stack project — chosen to be buildable end-to-end by one person and
to demonstrate engineering breadth (auth, persistence, security, tests, ops) without leaning on a
framework.

**HOW:** Deliberate zero-dependency constraint (later formalized as ADR-0001): pure Node `http`
module, a single-file SPA in `public/index.html`, and a JSON-file database (`db.json`). No npm
runtime deps, no build step.

**State at this point (approx):** core roles (student/org), opportunities, applications, hours,
awards, plans, leaderboard — the feature surface documented in `docs/FEATURES.md`.

## 2026-07-03 — Guardian consent flow + in-memory indexes  [reconstructed]

Committed as `8a1e4fe` ("Add guardian consent flow for minors and in-memory DB indexes", 12 files,
+1154/−80).

**WHAT:** (1) A guardian-consent gate for users under 18; (2) an in-memory index layer over the JSON
store.

**WHY (consent):** An `/llm-council` review (2026-07-02) concluded that verified guardian consent for
minors is a *precondition* for launch, not a feature — more urgent than any growth idea, because the
platform serves students and some are minors. Recorded as ADR-0010.

**HOW (consent):** Age is recomputed live from `dob` at every gate check (not cached at
registration), so 18+ users are never gated and a pending minor who turns 18 is unblocked
automatically with no background job. Single choke-point at `apply` (which transitively gates
messaging/check-in/endorsements, since those require an approved application), plus defense-in-depth
directly on those endpoints. Consent tokens are hashed (sha256) and never returned by any API — only
emailed. Public flow at `#consent/<token>` (approve/decline) and `#consent-manage/<token>` (revoke —
the kill switch). Correction mid-design: an earlier cut assumed *all* students were minors; that was
wrong (18+ students exist) and was fixed by making the gate age-dynamic and per-account.

**WHY (indexes):** Every lookup was an O(n) `Array.find`/`filter` over the JSON arrays; hot handlers
did several per request, and org dashboards/exports did N+1 scans. Pure waste since the keys are
known. Recorded as ADR-0011.

**HOW (indexes):** A lazily-built `IDX()` layer — `Map`s keyed by primary key and common foreign
keys — rebuilt on the *same* `_cacheVersion` invalidation the read cache already uses, so it can
never be staler than the last `saveDB()`. Invariant: never mutate the array `idxList()` returns
(shared bucket / frozen empty singleton) — spread first. `findIndex` write-path sites stay linear
scans (they need the array position). Covered by `test/index.test.js`.

## 2026-07-03 ~15:40 — Scaling cost optimizations (ADR-0012)  [contemporaneous]

**WHAT:** Seven server-cost optimizations aimed at keeping the bill flat as user count grows,
committed together after the editorial-restyle commit `10b0d05`. All backward-compatible (no response
shape changed); covered by a new `test/cost-optimizations.test.js` (7 tests) + ADR-0012.

1. **Coalesced writes (`saveDBSoon`)** — high-frequency low-value writes (view counts, notification
   read-flags) debounce (default 2 s) instead of a full-DB serialise per write.
2. **View beacon** — `POST /opportunities/:id/view` was doing `opp.views++; saveDB()` on *every page
   view*; now increments in memory + coalesces the write. This is the headline win.
3. **Notification polling (frontend)** — pauses while the tab is hidden (Page Visibility API) and
   refreshes on return, instead of hitting the server every 30 s per backgrounded tab.
4. **Static caching** — version-stamped assets (`?v=`) now get the same 1-year immutable caching as
   the emoji set (CDN-offloadable). Verified: `favicon-32.png?v=3` → `max-age=31536000, immutable`.
5. **Bounded notifications** — `retentionPurge` caps notifications per user to the most recent 200.
6. **Structural-signature index rebuilds** — `IDX()` now rebuilds only on DB swap or an *indexed*
   collection length change, not on every write. Safe because field mutations keep object references
   + FK groupings valid, and (verified by grep) the codebase never reassigns an id/FK in place nor
   replaces an array element in place. Decoupled from read-cache invalidation, which stays per-write.
7. **HTTP caching + N+1 fixes on public reads** — `jsonCacheable()` adds a cache-version-keyed weak
   ETag + short `Cache-Control` to the opportunity list / leaderboard / stats (304 on revalidation);
   the list's per-listing `applications.filter`/`opportunities.filter` and the leaderboard's
   per-student `hours.filter` now go through the `appsByOpp`/`oppsByOrg`/`hoursByUser` indexes.

**BUG found while doing #7 (good essay material):** the leaderboard endpoint called
`cacheGet('leaderboard')` but there was **no matching `cacheSet`** — so the in-memory cache was always
empty and the whole leaderboard (an O(students × hours) computation) was recomputed on every single
request. Root cause: the `cacheSet` wrapper was dropped at some point, leaving a dead `cacheGet`. Fix:
restore the `cacheSet` (15 s) and index the per-student hours lookup. So one endpoint was paying the
full cost on every hit while *looking* cached.

**WHY:** A code-review pass (requested by Evan) found several paths whose cost scales with traffic or
history rather than useful work — the things that inflate per-request CPU / disk I/O / RAM and thus
instance size at scale. See ADR-0012 for the full context/decision/consequences, including what was
**deliberately not** truncated (audit log, verified hours, messages — records/tamper-evidence) and
why the JSON-whole-DB model (ADR-0002) remains the real ceiling (`node:sqlite` is the next lever).

**HOW / verified:** `npm test` → 59 pass / 0 fail; `npm run coverage:check` → exit 0 (floor held);
live preview confirmed list still returns an array with badges+applicantCount, `/api/stats` returns
304 on a matching `If-None-Match`, leaderboard/stats carry `Cache-Control`, and the view beacon 200s.

## 2026-07-03 ~16:20 — Quantified the ADR-0012 wins + added `npm run bench`  [contemporaneous]

**WHAT:** Answered "how much time is saved at 10k users" with real measurements instead of a guess,
then made the measurement repeatable: new `scripts/bench.js` (`npm run bench`, `USERS=` to scale) —
a zero-dep, standalone micro-benchmark that builds a synthetic large corpus and times the exact
primitives ADR-0012 targets (full-DB serialise, index rebuild, leaderboard + opportunity-list reads,
old N+1 vs new indexed). It does NOT touch the real `db.json`. Folded the results into ADR-0012
(new "Measured impact" section) and this record.

**Numbers** (median, dev machine, 10k users / 76.7 MB DB — 10k users · 80k apps · 120k hours · 150k
notifs):
- `saveDB()` serialize + write: **~275 ms** each, and it *blocks the single-threaded event loop* —
  this is what the view beacon used to pay on every page view (now coalesced).
- full index rebuild: **~50 ms**, previously on every write (now only structural writes).
- leaderboard (uncached): **~11,800 ms → ~5 ms (~2300×)**.
- opportunity list (200): **~370 ms → ~1 ms (~400×)**.

**WHY this matters (the honest framing):** on a single-threaded server every one of those is
whole-process freeze time, not just the one request's latency. The old leaderboard alone (~11.8 s,
and recomputed every call due to the `cacheGet`-without-`cacheSet` bug) would have repeatedly frozen
the event loop at 10k users — an availability problem, not a latency one. A **modeled** daily
aggregate (~2,000 DAU; 40k views, 10k lists, 2k leaderboards/day) is ~10–11 h/day of blocking CPU
removed, but that's dominated by the leaderboard (partly a bug fix) and depends on the traffic
assumptions — so I recorded it as an estimate, not as a load-test result (`npm run loadtest` is the
tool for a real end-to-end figure).

**HOW / verified:** `npm run bench` runs clean and prints the table above; `node --check` /
`npm test` unaffected (bench is standalone, not imported by the server or tests).

## 2026-07-03 ~16:50 — Real end-to-end load test + the JSON-DB ceiling  [contemporaneous]

**WHAT:** Turned the micro-bench into a real HTTP measurement. `npm run loadtest` (the existing
tool) was dominated by the per-IP rate limiter (1,880/2,000 requests got 429'd from one IP — the
limiter working, not a capacity number), so I added `scripts/loadtest-scale.js` (`npm run
loadtest:scale`): spawns the real server against a synthetic large `db.json` with the limiter lifted,
drives the three optimized read endpoints, reports throughput + percentiles + single-shot latency.
Ran the same large DB against the pre-optimization `server.js` (commit `10b0d05`) for a true
before/after. Folded the numbers into ADR-0012.

**Numbers (same 10k-user / 59 MB DB, single-shot end-to-end):**
- `/api/leaderboard`: **17,293 ms → 0.3 ms**
- `/api/opportunities`: **4,907 ms → 38 ms**
- `/api/stats`: 6.2 ms → 0.4 ms
- Under load (current, 3k reqs @ 50): ~80 req/s, p50 ~620 ms, 0 errors. The old code would deadlock
  under any concurrency (a 17 s leaderboard request freezes the single-threaded loop for everyone).

**Two honest findings the load test surfaced (good essay material):**
1. `/api/opportunities` is now **response-size-bound**, not compute-bound — it returns *all* active
   listings uncached (~0.5 MB each; 1.5 GB moved across the run). Indexing fixed the CPU; pagination
   is the next lever, not more indexing.
2. **The architecture has a hard ceiling at ~90k users:** the whole DB is one JSON string and V8 caps
   a string at ~512 MB (10k users ≈ 59 MB → crossover ~90k). `USERS=100000 npm run loadtest:scale`
   fails outright — `Invalid string length` while writing `db.json` — so past ~90k the app can't
   write *or* load its DB and won't boot at all. Empirically confirms ADR-0002/0012; `node:sqlite` is
   the prerequisite to go further.

**WHY:** Evan asked for "a real number," then to fold it in and test at 100k. The 100k run is the
thing that exposed the string-length wall — a more valuable finding than a throughput figure.

**HOW / verified:** `node --check` on the new script; `npm run loadtest:scale` runs clean at 10k
(~80 req/s); `USERS=100000` reproduces the ceiling failure above. Standalone script — server + tests
unaffected.

## 2026-07-04 18:34 — SQLite migration + pagination (ADR-0013); the 100k ceiling is gone  [contemporaneous]

**WHAT:** Replaced `db.json`'s whole-JSON-file persistence with SQLite (`better-sqlite3` — one
collection per table, `id`/`seq` primary key + a `data` JSON-blob column per row), and added
`limit`/`offset` pagination to `GET /api/opportunities` (default 60, max 200, total count in
`X-Total-Count`). Scoped deliberately narrow: `loadDB`/`saveDB`/`backupSnapshot`/
`restoreFromBackup` are the only functions that changed — every `DB.<collection>` array and the
whole `IDX()` layer are untouched, since none of the 2,900+ lines of handlers cared what format
the data came from. New: `docs/adr/0013-sqlite-persistence.md`, `scripts/migrate-to-sqlite.js`
(one-time `db.json` → `db.sqlite` conversion, `npm run migrate:sqlite`). Updated `scripts/chaos.js`
and `scripts/loadtest-scale.js` (both used to read/write `db.json` directly, bypassing
`loadDB`/`saveDB` — had to teach them the new format).

**WHY:** ADR-0012's load test found the confirmed ceiling: `saveDB()`'s single
`JSON.stringify(DB)` call hits V8's ~512MB string-length cap at ~90k users
(`Invalid string length`, app can't boot past that point), and `/api/opportunities` was
response-size-bound (returns every active listing uncached). Evan authorized adding a real
dependency for this specifically ("using dependencies is ok, just preferably ones that are light
and easy to set up"). Chose `better-sqlite3` over the built-in `node:sqlite` after surfacing the
tradeoff: `node:sqlite` is zero-install but still experimental and only ships from Node 22.5+,
which would have dropped this project's Node 20.x CI leg; `better-sqlite3` is a stable, synchronous
API (fits the codebase's sync style) with prebuilt binaries, keeping Node 18+/20.x support intact.

**BUG found + fixed during this work:** the chaos harness's corrupt-DB scenario started failing
the graceful-shutdown check with `EPERM: operation not permitted, rename ... -> db.sqlite`. Root
cause: `_openSqlite()` opened a `Database` handle, then threw (on the first `CREATE TABLE`, not the
constructor — SQLite validates the file header lazily) when handed a corrupt/non-sqlite file,
without ever calling `.close()` on it. On Windows that left the corrupt file's handle open, which
blocked the recovery path's atomic rename-over-it. Fixed by wrapping the schema-creation loop in a
try/finally that always closes the handle before rethrowing. Caught by re-running
`npm run chaos` after the initial implementation — all 59 unit/integration/regression tests had
already passed without exercising this path, since none of them corrupt the DB file on disk.

**Numbers (`npm run loadtest:scale`, same synthetic-corpus methodology as ADR-0012):**
- 10k users, vs. the ADR-0012 (JSON-file, indexed) baseline: `/api/opportunities` single-shot
  38 ms → **3.9 ms**; throughput ~80 req/s → **1,445 req/s**; p50/p99 ~620/~820 ms → **34/46 ms**;
  response bytes moved across 3k reqs ~1.5 GB → **32 MB** (pagination).
- **100k users — the case that used to fail outright:** now completes cleanly. 763 MB DB,
  3,000/3,000 requests OK (0 errors), 130 req/s, `/api/opportunities` single-shot 19.4 ms. The
  confirmed `Invalid string length` ceiling is gone.

**HOW / verified:** all 59 tests pass (`npm test`), coverage floor holds (`npm run coverage:check`,
62/78/40% vs. 50/70/30% floor), `npm run chaos` 3/3 (including the corrupt-DB recovery scenario
that caught the handle-leak bug above), `node --check` on every changed file,
`USERS=100000 npm run loadtest:scale` completes without error. Full detail + the before/after
table: `docs/adr/0013-sqlite-persistence.md`. `.gitignore` updated for `db.sqlite`; `docs/`,
`CLAUDE.md`, `README.md`, `DEPLOY.txt`, `docs/disaster-recovery.md`, `CONTRIBUTING.md` updated
wherever they referenced the old `db.json` format as fact. ADR-0002 marked superseded (on-disk
format only — its in-memory-model reasoning still holds).

## 2026-07-04 19:20 — Security/perf/feature batch: WAL-incremental writes, gzip, page cache, password reset, check-in hardening, Discover URL state  [contemporaneous]

**WHAT (from a "find everything worth doing" audit, all seven items shipped together):**
1. **Incremental SQLite writes.** `saveDB()` no longer rewrites every row into a fresh file —
   a persistent WAL-mode handle + a per-row sha1 mirror (`_rowHashes`, ~20 bytes/row) means only
   changed/added/removed rows touch disk. WAL replaces the temp-file+rename dance for crash safety;
   the rename path survives as the fallback when no healthy handle exists (first boot / corrupt-file
   recovery — chaos scenario B still passes). `backupSnapshot()` checkpoints the WAL first so a bare
   file copy is a complete snapshot; `closeDB()` added to the shutdown path.
2. **Gzip for JSON responses** >1KB when `Accept-Encoding` allows (zlib, no new dep) + `Vary`.
3. **Whole-page compute cache** on `GET /api/opportunities` (version-keyed `cacheGet`/`cacheSet`,
   same pattern as the leaderboard) — repeat requests within max-age skip the filter/sort/badge
   pipeline entirely.
4. **Password reset flow** — the platform previously had NO way to recover a locked-out account.
   `POST /api/auth/forgot` (always-200, no account enumeration; sha256-hashed single-use token, 1h
   TTL; 3-req/15min throttle) + `POST /api/auth/reset` (weak-password checks, `tokenVersion` bump
   revokes all sessions, audited). Frontend: "Forgot password?" in the login modal, `#reset/<token>`
   route that drops the token from the URL/history immediately. `safeUser()` now strips the reset
   fields.
5. **Check-in hardening** — redemption was a full scan of every opportunity × every code with no
   guess limit. Now: O(1) `checkinCodeByStr` index (maintained at the generation site since codes
   are field mutations that don't fire the structural rebuild — index entries are hints re-validated
   against `opp.checkinCodes`) + per-user 10-fail/15min throttle.
6. **Discover URL state** — filters mirror into `#discover?q=…&cat=…` (replaceState, no history
   spam) and restore on load, so filtered views survive refresh and are shareable.
7. **"Showing X of Y"** count next to the Load-more button.

**Numbers (`npm run loadtest:scale`, same methodology):** 10k users: 1,445 → **4,539 req/s**,
p50 34 → **10 ms**, `/api/opportunities` single-shot 3.9 → **0.6 ms**. 100k users (763 MB DB):
130 → **4,167 req/s**, p50 371 → **11 ms**. The 100k case wouldn't boot at all two days ago.

**HOW / verified:** 66/66 tests (7 new: `test/password-reset.test.js` ×5, `test/checkin.test.js` ×2),
chaos 3/3, loadtests above. Gzip verified against a live server directly (the test seed is under the
1KB threshold, so the suite alone doesn't exercise it): gzipped body parses, `identity` requests skip
encoding. Full reset loop verified in the browser preview: forgot form → dev-log email link →
`#reset/<token>` form → old password 401 / new password 200. Discover URL round-trip verified in
preview ("Showing 2 of 2" on a filtered restore).

## 2026-07-05 — CI fix, then a graph-driven audit → 5 improvements  [contemporaneous]

**WHAT / WHY (CI, morning):** The SQLite migration made `better-sqlite3` a real runtime dep, but
CI's `coverage` and `resilience` jobs went straight from checkout to running their command with no
`npm install` — so they crashed on `Cannot find module 'better-sqlite3'` while the `test` jobs (which
do install) stayed green. Added the install step to both jobs; also bumped `actions/checkout` /
`setup-node` to `@v5` to clear the Node-20 deprecation warning. All 5 jobs green (`a7dda9f`, `5d3a852`).
Root cause was exactly the "CI will need updating once SQLite adds a dep" note left in HANDOFF —
the flag worked, we just hit it before reading it.

**Aside — the `console.log('` junk-file mystery, solved.** Stray 0-byte files kept appearing in the
working dir. Reproduced and traced it: `node -e "…"` one-liners containing an arrow function with
nested quotes (`j=>console.log(' ',…)`) *intermittently* get their `=>` split into `=` + `>` by a
shell layer in the Windows Bash-tool plumbing, so `>console.log('…` becomes an empty-redirect target.
Non-deterministic (~1 in 7 runs), always 0 bytes, never committed (staging is always explicit). Fix:
avoid `node -e` arrows — use `jq` or a temp `.js` file. Recorded in project memory.

**WHAT / WHY (audit batch, afternoon):** Ran the freshly-updated knowledge graph (`/graphify`, 639
nodes) as an audit lens. It surfaced five real items — each verified against the code before acting,
and two loud graph signals *rejected* as artifacts (namespaced doc-vs-AST duplicate nodes; a
meaningless "everything is 2 hops from a test" hub effect):

1. **`docs/security.md` was stale** — still listed password reset as "consciously deferred" though we
   shipped it 2026-07-04. Moved to the controls matrix (row 3b); later added MFA as row 3c.
2. **Notification email delivery (Track 2 #1)** — `createNotification()` wrote an in-app record but
   never emailed. The graph showed the write-side node and the `Real Notifications` roadmap node with
   no edge between them. Wired `sendEmail()` onto every notification, gated by a per-user
   `emailNotifications` opt-out (default on) exposed as a profile toggle for both roles. Test:
   `test/notifications-email.test.js` (captures the dev-stub log; asserts opt-out suppresses email
   but keeps the in-app record).
3. **TOTP MFA (RFC 6238), zero-dep** — new `lib/totp.js` (HMAC-SHA1, 6 digits, 30s, ±1 step drift,
   constant-time compare, base32). Enrollment: `mfa/setup` (pending secret + otpauth URI) →
   `mfa/enable` (live-code confirm; 8 one-time backup codes, stored sha256-hashed, shown once).
   Login became two-step: password → hashed single-use 5-min ticket → `mfa/verify` (throttled
   8/15min). Disable requires a valid code and bumps `tokenVersion`. `safeUser()` strips all five MFA
   fields. Frontend: settings section (both roles), login second-step form. Test: `test/mfa.test.js`
   ×6 (uses the real `totp()` to drive the flow — enrollment, two-step login, single-use ticket,
   backup-code consumption, throttle, secret hygiene).
4. **Extracted the SPA `<script>` to `/app.js` (ADR-0014, supersedes ADR-0007's inline rationale).**
   The graph produced 2 nodes for the whole ~3,100-line frontend because no AST could enter the inline
   script; the mirror of that opacity was CSP's `script-src 'unsafe-inline'`. Moved both inline blocks
   verbatim to `public/app.js` (`<script src defer>`, execution-order preserved), index.html 5,000 →
   1,884 lines. `'unsafe-inline'` still remains — the markup has ~270 inline `onclick=` handlers — so
   this is step 1; the CSP comment now states exactly what removes it, and a follow-up task is spun
   off to convert handlers to a dispatch table. **Did NOT bulk-convert them here** — that adds an
   argument-escaping layer with real XSS-regression risk and deserves its own focused change.

## 2026-07-05 ~21:30 — ADR-0014 step 2: delegated dispatch table → dropped `script-src 'unsafe-inline'`  [contemporaneous]

**WHAT / WHY:** Finished the follow-up that ADR-0014 step 1 deliberately deferred. The markup carried
~271 inline event handlers (`onclick=`, `onchange=`, `oninput=`, `onkeydown=`, `onblur=`,
`onmouseover=`, `onmouseout=`) across `public/index.html` (171) and the HTML template strings inside
`public/app.js` (100). Each is inline script to CSP, which is the only reason `script-src` still
carried `'unsafe-inline'` — neutralizing CSP as an XSS backstop for scripts. Converted every handler
to a single delegated dispatch table so the exception could be dropped.

**HOW:** One `ACTIONS` registry object + one delegated `document` listener per event type, both in
`app.js`. Markup carries a per-event attribute (`data-action` for click; `data-change`/`data-input`/
`data-blur`/`data-keydown`/`data-mouseover`/`data-mouseout` for the rest) naming a registry key;
arguments ride in a `data-args` JSON array; an optional boolean `data-stop` replaces the old
`event.stopPropagation();…` prefixes. Every entry has the uniform signature `(el, ev, a)`. Two
design points that mattered: (1) **blur doesn't bubble**, so its listener is attached in the capture
phase (verified the log-time `padNum` blur handlers still fire). (2) An element can carry a click
handler *and* a hover/keydown handler needing *different* args — since `data-args` is shared per
element, arg-carrying compound handlers were split into arg-free named entries (e.g. `padHr`/`padMin`
bake in min/max so the sibling `input` handler keeps `data-args` for its field key). Compound/
expression handlers (`nav('donate');loadDonate()`, `this.parentElement.style.display='none'`,
ternary-selected code strings, the onboarding `act` strings) became named registry entries computed
in JS *before* the template literal, never embedded as code.

**The XSS-critical bit (the whole reason step 2 was deferred):** arguments that used to be
interpolated into handler strings — including `openEditModal(${esc(JSON.stringify(o))})` passing a
whole opp object, and dozens of `'${jsq(o.title)}'` sites — now go into `data-args` as
`esc(JSON.stringify([...]))`. `esc()` escapes `' " < > &`, so JSON survives attribute parsing and
`JSON.parse` round-trips to the *raw* value the handler receives. Verified in-browser: a title of
`</div>"><img src=x onerror=alert(1)>'\` produced zero injected elements and an exact round-trip.
This escaping path **retired `jsq()` at every converted site** (its single-quoted-onclick-JS context
no longer exists). Kept the `jsq()` helper itself — it's referenced across 8 docs incl. append-only
ADRs, so removing it would cascade far beyond this change for a dead 2-line function.

Worked surface-by-surface (discover → student dash → org dash → admin → modals) with browser-preview
verification at each step: dispatch fires, the tricky dual `input`+`blur` log-time fields clamp+pad
correctly, the auth-modal flow (open → selectRole → switchAuth → close) works, footer nav + donate
amounts work. Then the gate — `grep` for inline handlers returns **0** in both files — and only then
dropped `'unsafe-inline'` from **script-src** in `server.js` (rewrote the CSP comment to explain the
split). `style-src` keeps `'unsafe-inline'`: hundreds of inline `style=` attributes are out of scope.

**Bug-watch during the work:** the browser served a **stale `/app.js`** on reload — it's sent
`Cache-Control: public, max-age=3600`, so within the freshness window the browser never revalidates
even though the ETag changed. This is a pre-existing gap the ADR-0014 split introduced (the doc is
loaded fresh but its unversioned `/app.js` subresource is cached for an hour → stale JS for up to an
hour after any deploy). Added a `?v=` version stamp to the `<script src>` (matching the repo's
existing `?v=` convention on `servelocal-logo.png` etc.), which also unblocked iterative verification.

**Verified:** `script-src 'self'` served with no `'unsafe-inline'`; page loads and dispatches with
**zero CSP violations** in the console under the locked-down policy; inline `style=` still applies
(style-src unchanged). `npm test` 74/74 green, `npm run lint` clean. ADR-0014 status → both steps
shipped.
